# 毕业设计选题调研：Vibe Coding 与 Agent 方向

> 调研日期：2026-07-14

## 零、结合学院材料后的最终判断

在读取《2027届数字媒体技术专业毕业设计选题方向指南》和《数字艺术学部2027届本科毕业论文（设计）工作方案》后，原先的“通用可验证编码 Agent”方向需要进一步绑定**真实行业场景、明确产品形态和数字媒体交互成果**。学院明确将“大语言模型、Agent、RAG、数字人、多模态交互”列入人工智能与数字人方向，但同时明确不建议“单纯调用 AI 接口、缺少自主开发内容”的项目。

### 最推荐题目

**《面向智能座舱 HMI 原型开发的可验证 Vibe Coding Agent 系统设计与实现》**

推荐理由：

1. **与学部重点方向高度一致**：智能座舱是2027届重点统筹方向，数字媒体技术学生被明确要求重点承担交互逻辑、系统原型、功能实现、数据调用、交互测试和技术文档。
2. **与专业指南高度一致**：同时覆盖“人工智能与数字人”“智能座舱与车载交互”“Web3D与数据可视化”三个专业推荐方向。
3. **不是简单 AI 套壳**：核心工作包括领域知识库、需求到原型代码生成、工具调用、运行沙箱、自动测试、HMI规范检查、版本追踪和可视化交互。
4. **成果可评价、可展示**：可以交付 Web 端 Agent 平台、可运行的车载 HMI 原型、座舱模拟数据、测试报告、演示视频、展板和源代码。
5. **有跨专业合作空间**：可与数字媒体艺术或视觉传达专业同学共享座舱项目背景，由对方负责视觉规范与动效，个人则独立完成 Vibe Coding Agent、原型生成链路和验证系统。

### 建议控制后的系统边界

用户用自然语言描述车载功能，例如“设计一个夜间长途驾驶场景下的疲劳提醒与音乐联动界面”。系统完成：

```text
自然语言需求
  → Agent 澄清用户、车辆状态与交互约束
  → 检索 HMI 规范和项目组件库
  → 生成任务计划与验收条件
  → 生成/修改 Vue3 HMI 原型代码
  → 接入模拟车辆数据
  → 自动运行功能测试、界面规则检查与安全检查
  → 根据失败证据自动修复
  → 输出可运行原型、代码差异、测试证据和说明文档
```

建议只支持 **Vue3 + ECharts + 可选 Three.js** 的 Web HMI 原型，不做真实车机系统、不训练大模型、不同时支持多种编程语言。这样既有完整工作量，也能在毕业设计周期内稳定完成。

- 题目必须体现“真实行业场景 + 核心技术 + 产品形态”，不能只写“Vibe Coding研究”或“Agent平台”。

- 最终成果要能现场演示或用视频稳定演示，并准备展板、二维码、技术说明等展示材料。

- 团队项目仍须一人一题，个人任务、成果边界、报告和评价依据必须独立。

## 一、结论摘要

若不考虑学院主题，优先推荐的是：**面向 Vibe Coding 的可验证软件开发 Agent：基于需求契约、自动测试与安全门禁的设计与实现**。结合学院材料后，建议将其具体化为上文的“智能座舱 HMI 原型开发”版本。

它比“做一个 AI 编程助手”更适合作为毕业设计，因为研究问题明确、系统成果完整、实验指标可量化，并且直接回应 Vibe Coding 当前最突出的问题：自然语言需求含糊、生成结果看似可用但缺乏可靠性、安全性和可维护性保证。

建议将系统定位为“Agent 质量控制层”，而不是重新训练大模型。用户提交自然语言需求后，系统将需求转为可验证契约，调用编码 Agent 生成或修改代码，再通过测试、静态分析、安全扫描和评审 Agent 形成反馈闭环，最终输出代码、证据链和质量报告。

## 二、方向判断

Vibe Coding 强调用户以自然语言描述意图、由模型完成大量实现；Agentic Coding 则进一步让模型自主规划、读取仓库、调用工具、修改代码并运行验证。当前前沿已从“能否生成代码”转向以下问题：

1. 如何准确理解含糊或不完整的用户意图；
2. 如何构造适合 Agent 工作的上下文、工具和反馈环境；
3. 如何用测试和可执行约束验证结果，而不是只相信文本回答；
4. 如何控制文件、命令、网络、密钥等操作权限；
5. 如何评估多轮、长时、跨文件的软件工程任务；
6. 如何让人能审阅、干预、追踪和复现 Agent 的行为。

因此，好的毕业设计应至少包含“Agent 自主决策 + 工具调用 + 验证闭环 + 可量化实验”，不宜只做聊天 UI、提示词封装或简单代码生成。

## 四、首选方案详细设计

### 4.1 建议题目

可从以下标题中选择：

- 《面向 Vibe Coding 的可验证软件开发 Agent 设计与实现》
- 《基于需求契约与多重反馈的智能编码 Agent 研究与实现》
- 《融合自动测试与安全门禁的 Agentic Coding 系统设计与实现》
- 《面向自然语言需求的软件开发 Agent 可靠性增强方法研究》

推荐使用第一个，名称直观且留有足够的实现空间。

### 4.2 核心流程

```text
自然语言需求
  → 需求澄清与结构化
  → 生成验收契约（功能、接口、边界、安全约束）
  → 仓库检索与任务规划
  → 编码 Agent 修改代码
  → 测试生成与执行
  → 静态分析 / 安全扫描 / 依赖检查
  → 评审 Agent 判断问题并反馈修复
  → 输出补丁、测试结果、风险说明和可追溯证据
```

### 4.3 系统模块

1. **需求契约模块**：把用户描述转换为结构化需求、验收条件和禁止事项；对关键信息缺失进行主动澄清。
2. **仓库理解模块**：建立文件、符号、依赖和测试索引，为 Agent 提供最小但相关的上下文。
3. **规划与执行模块**：将需求分解为任务，选择工具，编辑代码并运行命令。
4. **测试生成模块**：生成单元测试、接口测试和边界测试，并记录失败证据。
5. **质量门禁模块**：集成类型检查、代码规范、静态分析、依赖漏洞和密钥泄露扫描。
6. **反思修复模块**：根据验证失败结果定位原因并迭代，设置最大轮数和停止条件。
7. **权限与沙箱模块**：对读、写、执行、联网和敏感操作实施分级授权。
8. **可观测模块**：记录规划、工具调用、Token、耗时、修改范围、测试结果和人工干预。

### 4.4 可验证的研究假设

- H1：加入“需求澄清 + 验收契约”后，任务一次完成率高于直接提示生成。
- H2：加入“测试执行 + 静态分析反馈闭环”后，功能正确率和安全通过率显著提高。
- H3：仓库级检索与上下文压缩能够在基本保持成功率的同时降低 Token 成本。
- H4：按风险分级的人机确认机制能够减少危险操作，且不会显著降低任务完成效率。

### 4.5 实验设计

设置四组消融实验：

- A：直接让大模型生成或修改代码；
- B：A + 仓库检索与规划；
- C：B + 自动测试反馈；
- D：C + 需求契约、安全门禁与风险分级授权（完整方案）。

建议构造 40—80 个任务，覆盖：新增功能、缺陷修复、重构、接口联调、安全修复和含糊需求澄清。任务可来自自建的中小型 Web 项目与公开仓库，经人工校验后形成小型基准集。

主要指标：

- 功能正确率：通过隐藏测试的任务比例；
- 首次通过率与最终通过率；
- 安全缺陷数量及安全扫描通过率；
- 需求满足率与边界条件覆盖率；
- 修改范围准确性、无关文件修改数；
- 平均迭代次数、耗时、Token 与模型调用成本；
- 高风险操作拦截率、误拦截率；
- 人工干预次数和主观可用性评分。

统计上可使用配对实验；二元结果用 McNemar 检验，连续指标视分布采用配对 t 检验或 Wilcoxon 符号秩检验，同时报告效应量。

### 4.6 技术路线建议

- 前端：Vue 3 或 React，展示对话、任务计划、代码差异、执行轨迹和质量报告；
- 后端：Python FastAPI，负责 Agent 编排、任务状态和评测；
- Agent：采用支持工具调用的主流模型 API，模型层做成可替换适配器；
- 工具协议：MCP 或自定义 Tool API；
- 代码分析：Tree-sitter、语言服务器或基于符号的轻量索引；
- 验证：pytest/Jest、ESLint/Ruff、Bandit/Semgrep、依赖漏洞扫描；
- 隔离：Docker 容器、超时、资源限额、工作目录白名单和命令策略；
- 数据：SQLite/PostgreSQL 保存任务、轨迹、指标与实验结果。

建议优先支持一种技术栈，例如 Python + FastAPI 或 TypeScript + Node.js，不要一开始追求全语言支持。

## 五、两个高价值备选方案

### 5.1 Vibe Coding 安全治理 Agent

建议题目：《面向 Vibe Coding 应用的安全审计与自动修复 Agent 研究与实现》。

选择 OWASP 常见风险中的 5—8 类，例如 SQL 注入、XSS、鉴权缺失、越权访问、硬编码密钥、不安全文件上传和敏感信息泄露。系统执行“代码检索—风险判断—漏洞验证—补丁生成—回归测试”，并比较单 Agent、规则扫描、Agent + 静态分析组合方案的准确率和修复成功率。

优势是问题聚焦、数据和指标明确、社会价值强；若导师偏网络安全或软件测试，这是首选方案的最佳替代。

### 5.2 多 Agent 软件开发协同系统

建议题目：《面向仓库级软件开发的多 Agent 任务分解与冲突消解方法研究》。

设置规划、开发、测试、评审等角色，重点研究动态任务图、共享记忆、工作区隔离、并行修改冲突检测与合并策略。实验比较单 Agent、固定角色多 Agent、动态调度多 Agent在成功率、耗时、成本和冲突率上的差异。

优势是展示效果强、与产业趋势贴合；不足是范围容易失控。只有在开发周期充足、基础设施能力较强时才建议选择。

## 六、不建议直接采用的题目

- “基于 ChatGPT 的智能问答系统”：功能同质化严重，Agent 特征不足。
- “AI 代码生成平台”：范围过宽，难以提出清晰研究问题。
- “多 Agent 自动生成完整软件”：目标过大，验证标准不明确。
- “基于提示词优化的编程助手”：只比较提示词，系统工作量和论文深度通常不足。
- “训练一个代码大模型”：算力、数据、时间与可复现性风险过高。
- “接入多个模型进行代码生成对比”：容易成为产品拼装，需加入方法创新或系统性评测。

## 七、开题时应强调的创新点

创新点不要写成“使用了大模型、Agent 或 MCP”，这些只是技术手段。建议表述为：

1. 提出自然语言需求到可执行验收契约的转换与澄清方法；
2. 设计融合测试、静态分析和安全扫描的多源反馈闭环；
3. 提出基于风险等级的 Agent 工具权限控制与人机协同机制；
4. 构建面向真实仓库任务的可复现实验集与多维评测指标；
5. 通过消融实验验证各模块对正确性、安全性和成本的贡献。

## 八、阶段计划

1. 第 1—2 周：文献综述、需求分析、任务集与指标设计；
2. 第 3—4 周：完成基础编码 Agent、仓库工具和沙箱；
3. 第 5—6 周：完成需求契约、测试闭环与质量门禁；
4. 第 7 周：实现可视化界面和轨迹记录；
5. 第 8—9 周：构造任务集、开展对照和消融实验；
6. 第 10 周：分析数据、完善系统、整理案例；
7. 第 11—12 周：完成论文、演示视频与答辩材料。
