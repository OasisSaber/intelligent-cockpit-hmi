# 开发规范

## 1. 产品与技术事实的证据顺序

本节仅用于产品事实、实现事实或技术判断的证据取舍；它不定义通用工作流权威。工作流、安全、权限、受保护分支、发布和破坏性操作以根部 [`AGENTS.md`](../AGENTS.md) 为准。

1. 用户在当前任务中的明确决定；
2. 可运行代码、测试与实际工具输出；
3. 根目录和局部 `AGENTS.md`；
4. 当前项目进度、已接受的 ADR 与当前方向文档；
5. 早期报告、旧 Demo 和聊天摘要。

冲突时不得静默选择。记录冲突、采用证据更强的一方，并更新过时文档。

## 2. 架构边界

- Figma：设计 Token、组件、Variant、四屏稿和 Prototype；不承担运行时数据逻辑。
- React 前端：呈现、交互、动效、Web3D 和图表；不复制后端业务真相。
- FastAPI 后端：车辆状态、Profile、场景和多屏广播的权威来源。
- JSON：早期 Profile、场景和降级路线配置；MySQL 负责演示会话、风险事件与跨屏交互审计，不作为实时状态源。
- WebSocket：连续状态；HTTP：命令、配置和资源。

## 3. 前端规范

- TypeScript strict，不以 `any` 绕过领域建模；外部数据先校验再进入 Store。
- 组件按“领域含义”命名，不按临时视觉位置命名。
- 页面不直接散落车型判断；差异通过 Profile、Token 和组件 Variant 表达。
- 关键驾驶信息具有无 3D、无动画、无网络情况下的降级表现。
- 三维资源异步加载，避免阻塞仪表、HUD 和告警。
- 新交互至少覆盖成功、错误、加载、空数据和不可用状态中的适用项。

## 4. 后端规范

- Pydantic 模型定义跨层契约；接口变更同步更新前端类型和测试。
- 状态转换保持确定性；随机演示数据必须支持固定种子或可复现场景。
- Mock 与真实适配器实现同一接口，不让 UI 感知数据来源。
- API 错误使用稳定结构，不把堆栈或密钥返回前端。
- 长任务和外部调用设置超时、取消和可理解的降级结果。

## 5. HMI 与设计规范

- 安全相关信息优先于装饰性内容；HUD 只保留驾驶中必要内容。
- 告警至少使用图形、文字、位置或动效中的第二种通道，不只依赖颜色。
- 四屏共享设计语言，但信息职责不同，禁止简单复制同一页面。
- 目标是原创高端 HMI；可以学习 Tesla/SpaceX 的克制与系统性，不复制商标、图标、布局和资产。
- 每个 Figma 组件应能映射到代码组件、数据字段和可测试状态。

## 6. 文件与文档

- `materials/`：只读学校原件；
- `PreDesign/idea-archive/`：方向共识、案例和未决事项；
- `docs/adr/`：长期架构决策；
- `docs/project/`：当前决策、进度与实施路线；
- `deliverables/`：阶段报告和答辩交付；
- `outputs/`、`tmp/`：生成物和临时文件，不进入历史。

不得提交 `node_modules`、缓存、`dist`、`*.tsbuildinfo`、生成的 Vite 配置 JS/DTS、模型权重、可重新下载的大视频或任何密钥。

## 7. 验证矩阵

| 修改范围 | 最低验证 |
| --- | --- |
| 文档/提示词 | 路径、链接、术语、JSON 或 Markdown 结构检查 |
| 前端组件/状态 | 前端 lint、相关测试、前端 build |
| 后端模型/API | Ruff、后端 pytest；涉及运行态时 smoke |
| 跨层契约/依赖 | `pnpm check`，必要时 `pnpm smoke` |
| 性能/动画/Web3D | 构建 + 目标电脑手动帧率和降级验证 |

报告必须区分“命令通过”“手工验证”“未验证”。

## 8. 禁止事项

- 不得根据旧原型自动扩大毕业设计范围；
- 不得为追求技术数量引入无法验收的基础设施；
- 开始前检查仓库状态和相关项目文档；不得覆盖来源未确认的已有改动，发现工作区冲突时停止并请求用户决定；
- 不得在测试失败时删除测试或降低断言来制造通过；
- 不得未经授权推送、发布或改变 GitHub 可见性。
