## 1. 默认路由

Codex 在开始实现前先判断任务是否满足全部委派条件；满足时默认起草 OpenCode 合同，不先自行实现。

### OpenCode 候选

- 风险为 L0/L1，结果可由测试、构建、lint、快照或明确文件断言验证；
- 输入上下文可以列成最小文件清单，可写范围可以收敛到少量精确路径；
- 已冻结组件/布局的局部实现、样式修复、fixture、回归测试、机械文档；
- 已冻结接口后的 repository/query、检测插件或局部适配；
- 单一缺陷修复，且能够先写出稳定复现和非目标。

### Codex 保留

- 架构、课题范围、公共协议、共享状态权威、Token/设计系统；
- Agent 政策、合同起草与批准摘要、依赖和锁文件、CI、凭据与安全；
- 跨屏/跨层集成、模糊需求、性能诊断、发布、push、PR；
- OpenCode 结果的路径审计、技术裁决、独立验证和本地集成。

只按适合委派的任务计数评估约 65% 目标，不以拆碎任务或降低验收标准凑比例。

## 2. 合同与执行边界

每项新委派使用 `.opencode-subagent/drafts/<TASK-ID>.json` 的 schema v2 合同，并满足：

1. 一个合同只有一个有界结果、一个冻结的 Jujutsu base 和一组确定性验证；
2. `inputPaths` 是允许向外部模型提供的最小仓库上下文；
3. `allowedPaths` 是完整写入白名单，并必须包含结果回执；
4. OpenCode 在独立 sparse Jujutsu workspace/change 中运行，不接触主工作副本；
5. Supervisor 禁止 OpenCode 使用 Shell、Web、外部目录和子 Agent，并审计实际改动路径；
6. 新任务默认串行。只有 Codex 确认路径不重叠时才可双并发；
7. 草案对应的实现已完成、base 漂移或验收失效时，草案立即视为过期，不得派发。

`.opencode-subagent/config.json` 的 `protectedPaths` 在插件内置保护之上额外保护项目合同、Token、协作规范、旧编排控制面、插件源码和学校/历史材料。

## 3. 单次批准与外部数据共享

批准前，Codex 必须展示一份完整摘要，顺序固定为：

1. 结果与非目标；
2. 冻结 base revision；
3. 可写路径；
4. 将发送给外部 DeepSeek/OpenCode 的具体输入路径及数据类别；
5. 模型、运行时间、输入/输出 Token、成本和修正轮次上限；
6. 验证命令；
7. 自动本地动作与明确排除项。

批准指令固定为 `APPROVE <TASK-ID>`。只有在摘要已明确列出外部数据后，这一条指令才同时授权：

- 将列明的任务上下文发送给指定外部模型；
- 对冻结合同执行一次派发；
- Codex 在合同内复核、必要时接管修正；
- 独立验证通过后的本地 Jujutsu 集成。

执行命令必须使用 `approve --run --ack-external-data-sharing`，把授权写入合同和 runtime 审计记录。泛化表述如“批准了，下发”不得推断为外部数据共享许可；未展示数据清单或缺少确认标志时必须停止。这一规则用于修复此前 `TASK-0003` 因授权语义不完整而被平台安全策略拒绝的问题。

单次批准永远不包括：扩大路径、增加数据类别、换模型、加预算、再次派发、push、PR、发布或公开分享。以上变化需要新的明确授权。

## 4. P0：push 与 PR 外部发布门禁

Codex 不得把本地集成、任务批准或用户过去的 push 授权解释为新的外部发布许可。每一次 push 或创建 PR 都必须先提议并展示摘要，再由 Oasis 明确批准。

### push 摘要必须包含

- 远端 URL、已知的仓库可见性与信任状态；无法验证时必须明确写“未验证”；
- 将创建或更新的 bookmark/branch、顶端 change/commit 和全部尚未在远端的祖先 revision；
- 文件范围以及可能外发的源码、任务草案、runtime、日志、凭据或私人素材类别；
- 已运行的验证及未验证项；
- 明确说明不创建 PR、不发布、不删除或覆盖其他远端引用，除非本次另有批准。

批准指令固定为 `APPROVE PUSH <bookmark>`，且只授权摘要所列远端、revision 和一次 push。失败、平台阻止、远端变化或范围变化后不得自动重试，必须更新摘要并重新批准。

### PR 摘要必须包含

- repository、head、base、draft/ready 状态、标题和正文要点；
- 相对 base 的完整 diff 摘要、验证结果、已知风险和未完成项；
- PR 创建是否会通知外部协作者，以及是否包含图片、日志或其他可识别信息。

批准指令固定为 `APPROVE PR <head> -> <base>`。push 批准不自动包含 PR；PR 批准也不包含 merge、release 或后续 push。没有成功的远端回执时，handoff 不得声称已经 push 或创建 PR。

## 5. 默认预算档位

| 档位 | 适用任务 | 运行时间 | 输入 Token | 输出 Token | 成本上限 | 修正轮次 |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| S | 单文件/小型测试或样式修复 | 10 分钟 | 20,000 | 4,000 | 0.25 | 1 |
| M | 少量文件的有界实现 | 15 分钟 | 40,000 | 8,000 | 0.50 | 1 |
| L | 已冻结接口下的较大叶子实现 | 30 分钟 | 80,000 | 16,000 | 1.00 | 1 |

默认从 S 或 M 开始。达到 Token、成本或时间上限即进入 `BUDGET_EXHAUSTED`，不得自动追加；L 档仍不得承接 Codex 保留事项。

## 6. Codex 复核与失败处理

OpenCode 结束后停在 `AWAITING_CODEX_REVIEW`。Codex 只读取合同、结果回执、changed-path audit、预算记录、验证输出和必要 diff，避免重新加载全部实现上下文。

- 验证通过且无越界：记录 review，通过 Jujutsu 本地集成，再记录 integration receipt；
- 合同内小修可由 Codex 接管，不增加用户门禁；
- 越界、合同歧义或验收失效：不集成，保留现场并起草新合同；
- `STALE_BASE`、`ENVIRONMENT_DRIFT`、`BUDGET_EXHAUSTED`：不得自动 rebase、升级或加预算；
- push、PR、发布始终在本地集成之后单独请求授权。

## 7. Token 节流规则

- Codex 只为候选任务读取合同所需的最小上下文，不在主会话预先完成同一实现；
- 优先让 OpenCode生成实现与测试，Codex聚焦需求收敛、审计和失败诊断；
- 多个独立候选先形成短队列，逐项批准和执行，不把多个目标塞入一个超大合同；
- 交接记录委派候选数、实际委派数、复核结果和接管原因，用于调整 65% 目标；
- 若合同编写和复核成本已接近直接实现成本，任务保留给 Codex，并在 handoff 记录原因。
