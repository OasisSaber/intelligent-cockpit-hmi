# 开发规范

## 1. 事实来源优先级

1. 用户在当前任务中的明确决定；
2. 可运行代码、测试与实际工具输出；
3. 根目录和局部 `AGENTS.md`；
4. `CURRENT_HANDOFF.md` 与已接受的ADR；
5. 当前方向文档；
6. 早期报告、旧Demo和聊天摘要。

冲突时不得静默选择。记录冲突、采用证据更强的一方，并更新过时文档。

## 2. 架构边界

- Figma：设计Token、组件、Variant、四屏稿和Prototype；不承担运行时数据逻辑。
- React前端：呈现、交互、动效、Web3D和图表；不复制后端业务真相。
- FastAPI后端：车辆状态、Profile、场景和多屏广播的权威来源。
- JSON：早期 Profile、场景和降级路线配置；MySQL 负责演示会话、风险事件与跨屏交互审计，不作为实时状态源。
- WebSocket：连续状态；HTTP：命令、配置和资源。

## 3. 前端规范

- TypeScript strict，不以 `any` 绕过领域建模；外部数据先校验再进入Store。
- 组件按“领域含义”命名，不按临时视觉位置命名。
- 页面不直接散落车型判断；差异通过Profile、Token和组件Variant表达。
- 关键驾驶信息具有无3D、无动画、无网络情况下的降级表现。
- 三维资源异步加载，避免阻塞仪表、HUD和告警。
- 新交互至少覆盖成功、错误、加载、空数据和不可用状态中的适用项。

## 4. 后端规范

- Pydantic模型定义跨层契约；接口变更同步更新前端类型和测试。
- 状态转换保持确定性；随机演示数据必须支持固定种子或可复现场景。
- Mock与真实适配器实现同一接口，不让UI感知数据来源。
- API错误使用稳定结构，不把堆栈或密钥返回前端。
- 长任务和外部调用设置超时、取消和可理解的降级结果。

## 5. HMI与设计规范

- 安全相关信息优先于装饰性内容；HUD只保留驾驶中必要内容。
- 告警至少使用图形、文字、位置或动效中的第二种通道，不只依赖颜色。
- 四屏共享设计语言，但信息职责不同，禁止简单复制同一页面。
- 目标是原创高端HMI；可以学习Tesla/SpaceX的克制与系统性，不复制商标、图标、布局和资产。
- 每个Figma组件应能映射到代码组件、数据字段和可测试状态。

## 6. 文件与文档

- `materials/`：只读学校原件；
- `PreDesign/idea-archive/`：方向共识、案例和未决事项；
- `docs/adr/`：长期架构决策；
- `docs/ai-collaboration/`：Agent规范与交接；
- `deliverables/`：阶段报告和答辩交付；
- `outputs/`、`tmp/`：生成物和临时文件，不进入历史。

不得提交 `node_modules`、缓存、`dist`、`*.tsbuildinfo`、生成的Vite配置JS/DTS、模型权重、可重新下载的大视频或任何密钥。

## 7. 验证矩阵

| 修改范围 | 最低验证 |
| --- | --- |
| 文档/提示词 | 路径、链接、术语、JSON或Markdown结构检查 |
| 前端组件/状态 | 前端lint、相关测试、前端build |
| 后端模型/API | Ruff、后端pytest；涉及运行态时smoke |
| 跨层契约/依赖 | `pnpm check`，必要时`pnpm smoke` |
| 性能/动画/Web3D | 构建 + 目标电脑手动帧率和降级验证 |

报告必须区分“命令通过”“手工验证”“未验证”。

## 8. 禁止事项

- 不得根据旧原型自动扩大毕业设计范围；
- 不得为追求技术数量引入无法验收的基础设施；
- 不得在未读handoff时接管或覆盖已有change；
- 不得在测试失败时删除测试或降低断言来制造通过；
- 不得把模型自述作为完成证据；
- 不得未经授权推送、发布或改变GitHub可见性。

## 9. Agent编排标准

- 任务合同、runtime状态和最终回执分层保存；合同批准后以SHA-256封存。
- 新任务默认只设置一个用户监管门：一次`APPROVE`同时授权不可变合同的一次派发、列明的数据共享、Codex复核/接管和验证后的本地集成；内部状态迁移不得再次要求用户确认。
- 推送、PR、发布，以及范围、模型、预算、外部数据类别或权限边界的变化不属于上述授权。
- 默认串行执行；只有Codex明确证明文件范围独立时才允许最多两个workspace并行。
- 每个任务使用新会话，最多一次确定性纠错；异常中断不得自动重试。
- Supervisor必须独立于Codex Desktop运行，使用单实例lease和5秒心跳；90秒超时进入`INTERRUPTED`。
- 模型退出后由Supervisor使用结构化命令独立验证，模型自述不构成完成证据。
- 任务级sparse workspace只检出`inputPaths`与`allowedPaths`；Web、外部目录、Shell和sub-agent默认禁止。
- L0-L1可委派任务由Codex自动验证和复核；仅在合同实质变化或外部发布时再次请求用户授权。架构和关键基础设施永久为`CODEX_ONLY`。
- 所有真实模型执行前必须通过Fake Executor测试、环境锁、基线锁和预算预检。

### OpenCode模型策略

- `LOW`：局部、确定性强、修改范围窄；使用DeepSeek V4 Flash。
- `MEDIUM`：常规功能实现或多文件局部修改；仍默认使用DeepSeek V4 Flash。
- `HIGH`：需要较强跨文件推理但仍属于`DELEGATABLE`的任务；使用DeepSeek V4 Pro。
- 未声明复杂度时按`LOW`与Flash处理。复杂度由Codex在草案中决定，批准器负责选择并封存模型，Executor无权切换。
