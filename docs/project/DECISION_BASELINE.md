## 1. 课题定位

主标题保持为：**《智能座舱多屏协同 HMI 设计与交互原型》**。

副标题或摘要表述为：**基于共享状态与可扩展车载视觉感知的四屏协同设计**。

项目主线是高完成度的多屏 HMI 设计与全核运行系统，而不是单独证明某个 AI 接口、检测模型或 Web3D 效果。四个同等重要的端点为：

1. 主驾驶仪表；
2. HUD；
3. 中控屏；
4. 副驾驶屏。

四屏应具有同等级的设计深度和完成质量，但职责、信息密度和交互权限不同。

### 2.1 GP05 视觉冻结门槛

GP05 已通过视觉与交互审核，建议修补完成，进入实现阶段。该冻结不要求 Make 同时建立生产级 Variables、Component Set 或代码结构。冻结依据为以下两道门槛：

1. **风格门槛**：四屏各有至少一个代表性 Day/Night 主画面，字体、颜色、材质、间距、信息层级和动效语言一致；
2. **流程门槛**：导航接力、关键告警、副驾媒体与隐私三条流程可以在 Make 中完整演示，包含关键正常、告警、降级和恢复状态。

后续 React 完成后再进行 6—10 人的正式评价。视觉冻结停止条件为：四屏与三条流程完整、Day/Night 和关键告警成立、可读性与触控尺寸无阻断、资产可交付、现有架构和性能预算可实现、无未解决 P0。单纯“还能更炫”不构成重新开版理由。

### 2.2 Make 版本包与变更控制

- Make 候选版本必须使用唯一版本号和不可漂移的快照，不得只提供持续变化的链接；交付内容遵循 [`MAKE_VERSION_DELIVERY_TEMPLATE.md`](../design/MAKE_VERSION_DELIVERY_TEMPLATE.md)。

- 工程里程碑和代码 change 使用独立任务 ID，不复用 GP 设计编号充当提交版本。
- 冻结版本之后，文案、非关键间距等不改变合同的调整使用补丁版本；布局、导航、四屏职责、核心动效、Token 或三条流程变化属于实质变更，必须重新通过相关冻结门槛。

### 2.3 分层真相与实现偏差

- 冻结的 Make/Figma 快照是视觉意图真相；
- Token 与组件合同是设计到代码映射真相；
- FastAPI/WebSocket 是共享业务状态与行为真相；
- React 是最终运行表现真相，但不得无记录地偏离前三者。

Make 生成代码默认只作为视觉、交互、样式和局部行为参考，不整包并入生产前端。生产实现必须适配现有 React + TypeScript 结构、FastAPI/WebSocket 单一状态源、统一 Token、四屏权限、降级、可访问性、性能和依赖许可。目标逻辑画布追求高保真；响应式缩放、中文长度、真实数据、可访问性和性能引起的调整必须记录依据并通过截图与人工审核。

## 4. 四屏职责与权限

| 端点 | 核心职责 | 明确限制 |
|---|---|---|
| Cluster | 驾驶状态、路线状态、风险等级和风险处置状态 | 基本只读；无长文本、复杂设置和娱乐浏览 |
| HUD | 最低认知负荷的下一步驾驶决策与最高优先级行动提示 | 只读；无菜单、大图表、对话框和持续干扰动画 |
| Center | 路线规划、车辆/环境控制、风险详情与确认操作 | 不复制仪表全部信息 |
| Passenger | 娱乐、旅程协作输入、副驾安全反馈和允许的座舱控制 | 不拥有驾驶关键控制；不得用高亮动效干扰驾驶者 |
| Control | 选择场景、数据源、Profile 和检测器，查看诊断 | 是答辩控制台，不是车内产品端点；不能绕过 FastAPI 权威状态 |

驾驶者安全信息优先；驾驶中限制复杂操作。副驾可操作媒体、旅程、隐私和许可的座舱功能。HUD 不承载 AI 对话。

## 5. 三条核心跨屏工作流

### 5.1 导航任务接力

中控使用高德地图供应商能力完成地点检索和路线规划，确认后由仪表接管路线状态，HUD 提炼下一步转向，副驾参与旅程协作。`MapProvider` 适配层把供应商响应转换为统一路线模型，四屏不得直接依赖高德返回结构。

- FastAPI 调用高德路径规划 Web 服务；服务端 Key 和安全凭据只通过环境变量提供。
- 前端地图 JS SDK 只负责图源与可视化；Web Key 使用域名/来源限制，不与服务端凭据混用。
- 保存最近一次成功路线的标准化快照。供应商超时、限额或断网时明确显示服务不可用，并允许切换到已登记的本地答辩路线；本地路线只是降级证据，不冒充实时规划。
- 已开始的导航可继续使用已获取步骤；新的搜索和重规划在供应商不可用时进入降级状态。

### 5.2 关键告警升级与恢复

HUD/仪表优先显示，Center 提供详情和允许的操作，Passenger 抑制娱乐干扰，告警解除后四屏统一恢复。检测器不得直接决定界面，必须经过风险/策略层和权威状态。

### 5.3 副驾协作

Passenger 承担媒体、旅程协作输入和副驾安全反馈，Center 只呈现必要共享状态；娱乐信息不得进入 Cluster/HUD。风险发生时 Passenger 暂停娱乐并提示协助，但不得接管驾驶关键操作。

Profile 切换、主题同步和连接恢复是系统验证流程，不替代上述三条产品工作流。

## 6. Profile 深度

- `Executive SUV`：完整实现，作为主答辩 Profile。
- `Business Van`、`Performance Car`、`Track Mode`：只提供有证据的差异化适配，复用组件和共享状态，不复制整套流程。
- Profile 可以改变信息优先级、布局、功能、权限和动效，但不得形成互相矛盾的状态模型。

## 7. 运行时架构与协议

### 7.1 单一权威状态

FastAPI 是运行时唯一权威状态源。控制台发送命令，四个 React 端点通过 WebSocket 接收同一状态。客户端只保留输入焦点、临时弹层等瞬时 UI 状态；未确认命令不得被当作最终业务状态。

OpenCV Worker 只提交风险候选事件，不直接操作屏幕；MySQL 只负责持久化和审计，不作为实时状态源。地图供应商、Vision 检测器、测试注入器和答辩控制台都必须通过相同的命令/事件边界进入权威状态。

前端为一个包，固定路由：

- `/cluster`
- `/hud`
- `/center`
- `/passenger`
- `/overview`
- `/control`

答辩基线是一台高性能主机运行 FastAPI、MySQL、Vision Worker 和四个独立浏览器客户端。多显示器分别全屏投放；设备不足时使用 `/overview` 编排，但不得把四屏退化成一张静态页面。触控/鼠标是正式输入；键盘快捷键和控制台只用于答辩操作，并发送与正常 UI 相同的意图事件。

### 7.2 消息类型

协议只保留三类消息：`command`、`event`、`snapshot`。消息必须含协议版本、消息 ID、时间戳、来源、场景和 correlation ID。重连时先取最新完整 snapshot，再丢弃旧事件。

扩展域预留为 `energy`、`adas`、`driver`、`assistant`、`cabin`、`xr`，通过 feature flags/capability registry、端点 slot 和多模态事件协议扩展，不在界面放无意义的空卡片。

### 7.3 降级

- 每字段记录 fresh/stale/offline 与时间戳；stale 显示最后值和时间，offline 明确标识，不伪造实时数据。
- Cluster/HUD 显示安全最小值，Center 显示详情，Passenger 减少信息。
- 断线后通过完整 snapshot 恢复。
- 离线 Mock 必须明确标记为降级数据源，但仍能确定性完成答辩流程。
- 一键重置只结束当前演示会话、创建新 `session_id` 并恢复实时车辆/路线/风险/UI 初始状态，不删除历史数据库记录。

## 8. 主题、语言、可访问性与声音

- 环境状态自动控制 Day/Night，控制台可人工覆盖；四屏必须原子切换，只改变 Token/材质，不改变业务与布局状态。
- 手动覆盖在会话内保持；关键告警可以延后非必要主题动画。
- 中文为主要界面语言，汽车和品牌缩写可保留英文；使用 locale 字典，禁止把文案散落硬编码。GP04 英文只作为视觉基线。
- 正文对比度至少 4.5:1；大字和非文本控件至少 3:1。
- Center/Passenger 触控目标至少 48 × 48 逻辑像素。
- 关键状态同时使用图标和文字，不只依赖颜色；提供 reduced motion。
- 关键告警使用短促、不循环的 earcon；一般警告更弱，带冷却、媒体 ducking 和恢复。只预留触觉接口。

## 9. 性能与展示设备

目标设备为高端车型演示标准。当前主要验证机为 RTX 4070 Laptop，2560 × 1440 @ 144 Hz；另有 1920 × 1080 @ 144 Hz 输出。

| 指标 | 门槛 |
|---|---:|
| 四屏普通 UI | 目标 144 FPS；1% Low 不低于 100 FPS |
| p95 帧时间 | ≤ 10 ms |
| Web3D | 目标/最低验收 60 FPS；低于 55 FPS 必须调查 |
| 冷启动 | ≤ 2 s |
| 首个 snapshot | ≤ 500 ms |
| 跨屏同步 p95 | 目标 ≤ 50 ms；硬门槛 ≤ 150 ms |
| 关键事件到首个视觉响应 | 目标 ≤ 50 ms；硬门槛 ≤ 150 ms |
| 断线识别 | ≤ 1 s |
| 重同步 | ≤ 500 ms |
| Vision 推理 | 目标 20—30 FPS，且不得破坏 HMI 144 FPS 目标 |

144 FPS 只描述 HMI 交互和动效，不描述摄像头采集、模型推理或 Web3D。Web3D 是中控屏的车辆/座舱状态可视化模块，可响应车门、座椅、空调区域和停车守卫状态，但不参与导航、风险判定或共享状态核心逻辑；必须具有稳定 SVG/Canvas fallback。

性能结论只绑定冻结的答辩基准机。基准记录必须包含 CPU、GPU、内存、操作系统、浏览器、显示器刷新率、摄像头分辨率、模型与权重版本；144 FPS 只在实际 144 Hz 或更高输出上验收。当前参考设备为 RTX 4070 Laptop、2560 × 1440 @ 144 Hz，并提供 1920 × 1080 @ 144 Hz 输出。

## 10. VehicleVision 必选能力

VehicleVision 是通用车载视觉框架，不局限于疲劳监测：

- `vision-worker` 独立 Python 进程拥有摄像头、视频和模型；
- OpenCV 负责输入、预处理、帧调度、裁剪、基础跟踪和可视化；
- 疲劳/分心使用人脸关键点、眼部指标与头部姿态；停车守卫使用背景建模、运动区域与轮廓跟踪；乘员安全使用轻量目标/姿态模型与座位区域规则；
- MediaPipe、OpenCV DNN 或 ONNX Runtime 可作为 OpenCV 管线中的关键点、姿态和目标检测执行器；
- 检测器插件输出统一事件和 health，不直接修改 HMI；
- FastAPI 与 Vision Worker 不共享事件循环。视频帧使用有界队列，拥塞时丢弃旧帧而不是累积延迟；风险事件不得静默丢失；
- 风险/策略层处理独立阈值、连续帧/持续时间、解除条件、冷却和升级，再写入 FastAPI 权威状态；
- 风险生命周期固定为 `candidate → active → acknowledged → resolved`。同一持续事件只更新记录，不重复创建；四屏只响应 `active` 及后续状态；
- worker 崩溃不得导致 HMI 退出；不引入 MQTT、ROS2、Docker 或额外中间件；
- 本地推理，不保存现场原始画面，不跨会话做人脸身份或持续追踪。

首批三个场景：

1. 驾驶员疲劳/持续闭眼与分心：现场摄像头；
2. 停车守卫模式的运动/人员接近：现场摄像头；
3. 乘员手机使用、人员离座或大幅越界：授权预录视频，但必须经过与现场输入相同的真实推理管线。

首版不在缺少合适模型与数据集时宣称可靠识别安全带。排除人脸身份、情绪推断和法律行为定性。

### 10.1 Vision 验收

- 每个场景具有独立正例、负例和边界用例；调参集与测试集分离。
- Precision ≥ 90%，Recall ≥ 85%。
- 负例误报 ≤ 1 次/10 分钟。
- 普通行为事件延迟 ≤ 300 ms；疲劳使用持续时间规则。
- 视频和现场摄像头分别验证。
- 摄像头、模型或低置信度异常必须可见降级。

控制台显示标注视频、置信度、FPS 和时间线；驾驶中的 Cluster/HUD 不显示视频。Center 只在停车、诊断或守卫详情中显示，Passenger 只显示与自身相关的安全提示。

视觉来源必须明确标记：`LIVE CAMERA`、`VIDEO INFERENCE`、`SIMULATED EVENT`。手动注入只能验证后续联动，不得冒充真实识别。

现场由 Oasis 本人操作，不增加知情同意、身份管理或影像授权模块。默认不保存连续原始视频；风险触发时可配置保存一张本地事件截图，数据库只记录相对路径。截图可关闭，并可由显式开发命令清理；一键重置演示不会删除历史事件。

仓库只提交模型配置、版本、来源、许可证、SHA-256 和安装脚本，不提交大体积权重。答辩机器必须提前完成离线缓存；运行时缺失模型时明确进入降级状态，不临时联网下载。预录视频同样登记来源、用途和哈希，私人素材不得进入版本库。

## 11. MySQL 与全栈边界

MySQL 是必选的持久化和审计层，但不是 144 FPS 实时状态源。

应持久化：

- 演示会话；
- OpenCV 风险事件及其生命周期；
- 跨屏交互/命令审计记录。

不得持久化：每个遥测帧、144 FPS UI 状态、原始摄像画面、WebSocket 瞬时状态、模型缓存和权重。

风险事件稳定核心字段为：事件 ID、会话 ID、来源、事件类型、严重等级、置信度、发生时间和处理状态。场景特有信息进入 `metadata JSON`，例如座位区域、目标类别、持续时间、边界框、事件截图相对路径、模型版本和停车守卫运动区域；高频检索字段不得全部塞入 JSON。

技术栈：MySQL 8.x 原生服务、SQLAlchemy 2 async、`asyncmy`、Alembic、Pydantic API 模型与 ORM 模型分离、repository/service 层、批量异步写入、确定性 seed。测试使用真实 MySQL 测试库，不以 SQLite 冒充。

实时风险先进入 FastAPI 权威状态并广播四屏，再由有界持久化队列写入 MySQL。失败时标记 `persistence_pending`、有限重试并在诊断面板报错；数据库恢复后补写，但数据库异常不得阻断四屏告警，也不得无限堆积内存。

本地使用低权限应用账户；只提供环境变量名示例、PowerShell/SQL 初始化、health、迁移和备份恢复文档，不提交密码。项目不实现通用注册、登录、JWT、OAuth 或短信；使用预设 persona 和本地会话。

## 12. 数据来源、可观测性与隐私

- 导航正式数据来自高德 `MapProvider`；本地确定性路线只作供应商失败时的降级。遥测和非 Vision 告警由确定性场景引擎生成。
- `/control` 与 `/overview` 显示数据源、场景名、运行时间及 `SIMULATED`/`LIVE CAMERA`/`VIDEO` 标签；四个产品端不重复加模拟水印，启动页或演示说明统一声明。
- 每次演示生成唯一 `session_id`，统一记录命令、snapshot、跨屏事件、视觉判断、数据库状态、FPS 和延迟。
- `/control` 诊断面板实时显示四屏连接、共享状态版本、WebSocket 延迟、UI FPS/1% Low、Web3D FPS、OpenCV 推理 FPS、当前模型、事件状态机、MySQL 写入结果和地图供应商状态；指标必须来自真实运行数据。
- 时间线支持回放，并导出脱敏 JSON/CSV。
- 日志移除本机路径、设备标识和身份信息；用户测试只记录匿名任务与问卷数据。
- 视频、模型和数据集不进入源码历史；仓库只保存 manifest、来源、许可证、哈希、说明和少量安全 fixture。
- 开发/测试模式可启用风险事件注入器，通过与 OpenCV 相同的事件接口覆盖四屏、数据库和恢复测试；正式演示必须区分真实识别与测试注入，生产构建默认禁用注入入口。

## 13. 加分项与范围门槛

预留能力池：

1. 疲劳监测与自适应 HMI；
2. AI 语音助手与跨屏任务接力；
3. 新能源汽车数据可视化；
4. 数字人、手势、情绪座舱和 XR 仅保留兼容接口。

VehicleVision 是当前唯一重点加分方向，三个场景按第 10 节分级验收。XR 不进入当前实现范围，只保留兼容接口。AI 车载助手只预留接口与界面位置；核心、Vision、数据库和性能门槛全部通过后，才可评估“语音指令触发已有车辆操作”，不做独立数字人、情感模型或复杂 LLM 自主决策。

AI 若启用，只能发送受约束的 FastAPI command：查询可直接回答；低风险座舱/媒体操作执行后反馈；目的地、隐私、座椅等中风险操作需要确认；安全关键操作永不直接执行。离线固定意图必须可用，在线 AI 只作增强。

## 14. 验证与评价

四层测试：

1. 协议：消息 schema、版本、顺序、重连和旧事件丢弃；
2. 状态：三条工作流、权限、主题、告警和降级；
3. 组件：中文长度、主题、可访问性和状态覆盖；
4. 视觉/E2E：四画布、GP04 回归、目标距离可读性和性能。

任何能力只有同时具备可运行实现、自动化测试或可重复测试步骤、以及答辩可展示证据，才允许标记完成。只有页面、预设动画、接口占位或模拟结果的不算工程实现完成。最低证据包括：四客户端同时在线、三条流程可重复执行、摄像头与视频真实推理、风险事件实际写入 MySQL、性能实测、断网/摄像头/Web3D 降级，以及一键重置后的完整重演。

用户评价分两轮：视觉冻结前使用 Make 做小规模形成性测试，发现信息层级、跨屏理解和流程问题；React 完成后使用 6—10 人完成三条工作流的正式评价，记录完成率、时间、错误、跨屏理解，以及可读性、一致性、信任和负荷 Likert 评价。至少保留一项前后对比。Vision 指标和性能报告独立呈现，不用主观问卷替代。

## 15. 实施顺序与冻结线

实施依赖顺序固定为：

1. 本决策基线、GP04 与 `GP04-CODEX-STRUCTURED-v0.1` 归档；
2. Make 主导 GP05 视觉与交互探索、形成性测试和建议修补，Oasis 批准视觉冻结；
3. 提取 Token，冻结组件合同、公共协议、逻辑画布和数据库核心表；
4. FastAPI 权威状态、WebSocket、场景引擎与四端路由骨架；
5. 三条核心跨屏流程；
6. 高德 `MapProvider` 与本地降级路线；
7. MySQL 会话、风险事件和交互审计链路；
8. VehicleVision Worker、摄像头/视频输入和三类检测场景；
9. Web3D 状态可视化与 2D fallback；
10. 诊断、故障恢复、性能与自动化测试；
11. Profile 差异、正式用户评价、录制、报告和归档；
12. 仅在全部核心门槛通过后评估受约束的 AI 语音助手。

四道冻结线：

- **视觉冻结**：GP05 通过风格与流程门槛并由 Oasis 批准；实质视觉变化必须重新评估影响。
- **结构冻结**：Variables、基础组件、公共协议和数据库核心表确认后，变更必须记录影响。
- **核心冻结**：四屏三条工作流通过后，只修缺陷和性能，不扩展核心范围。
- **答辩冻结**：证据包录制后，只处理阻断演示的问题，不升级依赖、不重构架构。

桌面封装只在核心稳定后评估；浏览器路由、多显示器、overview 和一键启动是基线，不能依赖 Electron/Tauri 才能答辩。

按答辩日期倒排：前三周完成核心功能闭环，前两周功能冻结且只修缺陷/性能，最后一周只做数据准备、部署复现和答辩演练。未在功能冻结前达到同等完成定义的附加功能不得进入现场主流程。
