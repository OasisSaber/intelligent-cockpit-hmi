## 2. 学校要求摘要

- 总主题为“智艺·新境”，强调智能技术与艺术创意融合；
- 智能座舱与车载数字体验属于重点方向；
- 数字媒体技术个人任务应落到交互逻辑、系统原型、功能实现、数据、测试和技术文档；
- 坚持一人一题、个人成果边界清楚；加入跨专业团队后需重新备案个人题目、分工和成果；
- AI 可辅助但不得替代个人工作，提示词、修改、测试和采用情况需要留痕；
- 成果需可评价、可展示、可归档，并准备离线展演、源码、演示视频和技术说明；
- 题目必须具体，范围和工作量合理，不能只做简单网页或套用 AI 接口。

逐项来源、章节、已确认页码、风险和待确认事项见 `docs/01-school-requirements-mapping.md`。

## 7. Demo 已实现功能

- 车辆状态模拟器持续生成车速、挡位、电量、续航、驾驶模式、电池温度和安全带状态；
- 汽车座舱风格数字仪表、中控 HMI、驾驶员状态、道路区域、事件时间线和证据区；
- 本地视频文件选择与浏览器内播放，无文件时保持 Mock 离线画面；
- 从 `demo-data/mock/events.json` 加载五个模拟感知帧；
- Pydantic 统一道路、驾驶员、车辆、风险事件和行程记录结构；
- 显式规则引擎输出低/中/高风险；
- “前方行人 + 驾驶员分心”生成 `pedestrian_and_distraction` 高风险事件；
- FastAPI HTTP API 与 WebSocket 实时推流；
- HMI 风险颜色、提示、驾驶员状态、道路标签和证据动态联动；
- 时间线按事件类型和时间戳去重，避免开发模式重连产生重复记录；
- 结构化行程记录与离线 Mock LLM 风险解释、行程报告；
- 无模型、无 GPU、无 API 密钥、无网络时仍可运行。

当前没有将 Mock 检测描述为真实模型能力。真实车辆/行人/车道和疲劳视觉推理、语音、真实 LLM 工具调用及视频时间点跳转均在文档中标为后续工作。

## 11. 下一阶段建议

1. 先与指导教师冻结题目、P0 范围和个人成果边界；
2. 选择一段授权明确的城市通勤短视频和一段本人授权的座舱视频；
3. 按“OpenCV 读帧 → 预训练检测 → 状态适配器”的顺序接入道路感知；
4. 接入 MediaPipe 驾驶员关键点，先完成闭眼、打哈欠、头姿/视线偏移基础规则；
5. 用场景清单验证复合风险规则，再决定是否接入真实 LLM 和语音；
6. 从第一次真实数据实验开始填写 AI 使用记录和数据许可记录，所有指标只写实测值。
